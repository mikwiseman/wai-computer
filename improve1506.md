## 1. Executive summary

WaiComputer has built the *hard* half of a category-defining product and shipped the *easy* half of a lesser one. Underneath is genuinely rare second-brain machinery: cited "ask your brain" answers across recordings, notes and chats; unified RRF hybrid search with reranking; compiled-truth dossiers; a bi-temporal `EntityFact` graph; a nightly sleep-time consolidator; and — uniquely — a *hosted* MCP endpoint that lets any AI agent read and (opt-in) write memory. The competitor sweep is unambiguous: cross-source cited Q&A and agent-native (MCP) access are exactly the capabilities the 2026 field is racing toward, and Wai is ahead of it. Yet almost none of that value reaches a user.

Three problems sit on top of the moat.

**The moat is unplugged and now UI-less.** Verification surfaced the decisive fact: the Brain UI — and the connect-any-data-MCP *ingestion* feature — were *deliberately removed across all clients* on 2026-06-10 (commit `1d9b9258f`, "remove MCP ingestion connectors and Brain UI across all clients"). The backend recall machinery survives (the `ask`/`search` API, RRF hybrid search, compiled dossiers), but it is now **headless** — reachable only through the MCP tools, with no first-party UI on web, macOS, or iOS. The `EntityFact` bi-temporal store is dead end-to-end (`entity_graph.py`), and Companion "Ask Wai" is regex-routed, not a real cited-recall agent (`wai_agent.py:303`). This lands one day after the `BRAIN_REDESIGN.md` cockpit→calm study, so it reads as a teardown-to-rebuild — but **as of today the thing that makes Wai special has no surface a user can open.**

**Silent failure is the most common defect — directly against the stated "no fallbacks" ethos.** iOS has *no background-audio capability*, so a recording dies on screen-lock or app-switch (CRITICAL). Android loses an in-flight recording if the process is killed, swallows errors, and crashes on an unhandled `delete()`. Backend stale-recovery orphans staged audio; Swift's `copy(buffer:)` returns the live aliased buffer on allocation failure. Data is lost or corrupted while the UI reports success.

**"Six platforms, one memory" is not true yet.** Windows is a static mockup — no ViewModels, no `DataContext`, a referenced-but-undefined `DpapiSessionProtector` (multiple CRITICALs). Linux is a capability-probe shell with no recording or auth. iOS ships dead placeholder Settings wired to keys nothing reads. The differentiator ships on macOS and only partially elsewhere.

The through-line: code quality, UX, and the agent-memory wedge are one problem. A brain that hides its best feature, loses data while claiming success, and only half-exists across the platforms it advertises is a recorder with good intentions. Plug the recall in, surface it as a cited answer in every app, make capture trustworthy, and make the MCP moat loud — and Wai becomes the second brain that actually remembers.

## 4. Findings by surface

### Shared Swift core (WaiComputerKit) — audio capture, realtime/dictation streaming, session/auth, monitoring

_WaiComputerKit is a ~30K-line shared Swift core (plus ~tests) that powers native live recording, realtime transcription, and system-wide dictation across macOS/iOS. The architecture is mature and clearly battle-scarred: actors serialize the streaming state machines, there is a strong local-first recording backup store with explicit failure states (no silent data loss), a thoughtful CATap system-audio path with silent-failure verification, and a genuinely good privacy sanitizer for Sentry. However there are two realtime streaming paths that have DIVERGED: the newer dictation path (DictationSession → ProviderBackedRealtimeSession) correctly awaits the WebSocket handshake via WebSocketHandshakeCoordinator (the fix for the lost-wakeup P0), while the live-recording path (WebSocketManager, used by both Mac and iOS RecordingViewModel) never adopted that fix and still yields .connected optimistically. There are also several real correctness/perf issues in hot audio/SSE paths (O(n²) SSE parsing, per-sample PCM Data.append, unstructured Task per audio buffer with no ordering guarantee, an asymmetric dual-capture flush that loses/leaks on mic stall) and a token-at-rest accessibility-class issue on iOS. Findings below are all grounded in code I read._

_4 high, 6 medium, 3 low · 13 findings_

#### [S1-1] `HIGH` · architecture · effort L — Live-recording WebSocket path never adopted the handshake-await fix (same class as the dictation lost-wakeup P0)
- **Where:** shared/WaiComputerKit/Sources/WaiComputerKit/Network/WebSocketManager.swift:192-209 and :413-417 (connect/receiveMessages); contrast ProviderBackedRealtimeSession.swift:95-125
- **Problem:** WebSocketHandshakeCoordinator was built specifically so a caller can await the real HTTP→WS upgrade and catch a close (e.g. proxy 1008/1011 on a bad/expired token) that arrives during the handshake window. It is wired ONLY into the dictation path (ProviderBackedRealtimeSession). The live-recording path is WebSocketManager, instantiated by BOTH MacRecordingViewModel.swift:386 and iOS RecordingViewModel.swift:191. WebSocketManager.connect() calls socket.resume() and immediately starts receiveMessages(), which yields .connected (WebSocketManager.swift:417) before the upgrade completes. A close during the handshake therefore surfaces only as a generic receive() error that triggers the auto-reconnect loop (reconnectEnabled=true), so on a structurally bad token the recording shows no live transcript while silently churning up to 10 reconnect attempts. This is the exact failure mode the coordinator exists to prevent; the recording path was simply left behind. Impact is bounded (recording is local-first, so audio is still saved to disk and batch-transcribed), but live transcript reliability and reconnect noise are user-visible.
- **Fix:** Route live recording through the same handshake-aware open() (either reuse ProviderBackedRealtimeSession for recording, or give WebSocketManager a WebSocketHandshakeCoordinator + a dedicated delegate URLSession and await waitForOpen before yielding .connected). Then distinguish a handshake-time auth/close failure (do NOT auto-reconnect; surface the error per the no-fallback rule) from a mid-stream drop (reconnect). Consolidating the two near-duplicate Deepgram WS implementations would also remove a large maintenance hazard.
- _confidence: high_
- **Verified:** ✓ confirmed _(adversarial re-read)_ (loc: shared/WaiComputerKit/Sources/WaiComputerKit/Network/WebSocketManager.swift:195-203 & :413-417; contrast Network/ProviderBackedRealtimeSession.swift:95-125)

#### [S1-2] `HIGH` · performance · effort S — SSE parsers rescan the entire pending buffer on every byte (O(n²) per frame) on the dictation + companion hot paths
- **Where:** shared/WaiComputerKit/Sources/WaiComputerKit/Network/DictationCleanupStream.swift:150-167 & :209-228; identical in Network/CompanionStream.swift:327-340 & :385-407
- **Problem:** Both SSE decoders iterate the network stream one byte at a time (`for try await byte in bytes`, fed by URLSession.bytes(for:) at APIClient.swift:1737/1901/1963) and after appending each byte call firstFrameTerminator(), which does `let bytes = [UInt8](self)` — a full copy of the whole accumulated buffer — then linear-scans it for \n\n. For a frame/response of N bytes this is O(N²) work plus O(N²) transient allocations. Dictation cleanup is on the latency-critical paste path (memory notes cleanup p95 ~5s); the companion/Ask stream can be long. This burns CPU and allocator pressure exactly when the user is waiting for text.
- **Fix:** Scan incrementally: track a search cursor so each new byte only checks the last 1–3 bytes for the terminator, and avoid materializing `[UInt8](self)` per byte (scan the Data directly or use `bytes.lines`/a chunked AsyncBytes wrapper). Reuse one helper for both parsers. This is a localized, well-testable change with existing test files (CompanionStreamTests) to guard it.
- _confidence: high_
- **Verified:** ✓ confirmed _(adversarial re-read)_

#### [S1-3] `HIGH` · logic · effort M — Audio tap callback spawns one unstructured Task per buffer with no ordering guarantee — PCM frames can be reordered
- **Where:** shared/WaiComputerKit/Sources/WaiComputerKit/Audio/AudioEngineHost.swift:82-90 (installTap closure) and handleTapBuffer at :214-225
- **Problem:** The real-time tap closure does `Task { await self.handleTapBuffer(copy) }` for every captured buffer. Each call creates a separate top-level Task; Swift makes NO FIFO guarantee about the order in which independently-created Tasks reach an actor's executor. Under load (or just normal scheduling jitter) buffer K+1 can be appended to the pre-roll ring and yielded to the live AsyncStream before buffer K, scrambling the audio timeline that is then sent to Deepgram. This degrades dictation/transcription accuracy intermittently and is very hard to reproduce. The actor protects against data races but does nothing for ordering across distinct Tasks.
- **Fix:** Preserve submission order: feed buffers through a single serial conduit — e.g. an AsyncStream the host drains in one `for await` loop, or an actor method invoked from one long-lived consumer Task — instead of N independent `Task {}`s. (The same per-buffer `Task {}` pattern should be audited anywhere else tap callbacks hop onto an actor.)
- _confidence: medium_
- **Verified:** ✓ confirmed _(adversarial re-read)_

#### [S1-5] `HIGH` · security_privacy · effort S — iOS auth tokens stored with kSecAttrAccessibleAfterFirstUnlock (not ...ThisDeviceOnly) — tokens migrate via device/iCloud backup
- **Where:** shared/WaiComputerKit/Sources/WaiComputerKit/Auth/KeychainHelper.swift:19; consumed for tokens at ios/.../App/WaiComputerApp.swift:328-329,411-414,437-440,506-509
- **Problem:** KeychainHelper saves with kSecAttrAccessibleAfterFirstUnlock. That accessibility class is included in encrypted device backups and iCloud Keychain, so the access+refresh tokens can be restored onto a DIFFERENT device. iOS still uses KeychainHelper as its primary token store (the file-based SessionStore is macOS-only; on iOS Keychain survives reinstall just fine). For a product that is the user's entire second brain (recordings, transcripts, memory), refresh tokens leaving the device via backup is a real account-takeover surface.
- **Fix:** Use kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly for the access/refresh token items so they never leave the device. (The macOS Keychain path is being phased out in favor of SessionStore, but iOS still depends on this.) Verify existing items get re-saved with the new class on next login.
- _confidence: high_
- **Verified:** ✓ confirmed _(adversarial re-read)_

#### [S1-4] `MEDIUM` · logic · effort M — DualAudioCapture flush is asymmetric: a mic stall (with system audio still flowing) loses sync and grows systemBuffer unbounded
- **Where:** shared/WaiComputerKit/Sources/WaiComputerKit/Audio/DualAudioCapture.swift:272-320 (flushDualBuffers)
- **Problem:** flushDualBuffers has explicit handling for a SYSTEM-audio stall (mic with silence padding, branch at :288-291) but the inverse is unhandled: if `micCount < minFlushSize` the function returns early at :295-299 regardless of how much system audio has accumulated. If the mic input stalls mid-recording (route change, device disconnect, AirPods drop) while system audio keeps delivering, systemBuffer (a `[Float]`) grows without bound — a memory leak proportional to recording length — and nothing is flushed, so the live/recorded stream stalls even though one source is healthy. The stall detector only watches system audio (zeroSystemCount), so a mic stall isn't even surfaced.
- **Fix:** Make flush symmetric: add a branch that flushes system audio padded with mic silence (capped, mirroring the existing system-stall branch) when system has >= minFlushSize but mic is stalled, and add mic-side stall detection. Per the no-fallback rule, also surface a mic-stall warning to the user rather than silently degrading.
- _confidence: medium_

#### [S1-6] `MEDIUM` · product · effort M — AVAudioEngineConfigurationChange ends an active dictation turn silently instead of hot-recovering
- **Where:** shared/WaiComputerKit/Sources/WaiComputerKit/Audio/AudioEngineHost.swift:129-156 (handleEngineConfigurationChange) — see the docstring at :108-115 'Full hot-recovery ... is a follow-up'
- **Problem:** When the HAL renegotiates (AirPods connect/disconnect, USB mic plug, default-input switch) macOS auto-stops the engine and drops the tap. The handler correctly finishes the live continuation, clears the lease, and resets preWarmed so the NEXT dictation re-warms — but it does not restart the engine or resume the in-progress turn. So if a route flips while the user is mid-dictation, the turn ends with only the audio captured before the flip; the user gets a truncated transcript and no signal that anything went wrong. On Tahoe (which the codebase notes fires this 2–5× more), this is a realistic mid-sentence event (e.g. AirPods auto-switching).
- **Fix:** Implement the deferred hot-recovery: if a lease is active when the engine stops, re-prewarm and re-install the tap, then either continue the same DictationSession or finalize-and-surface a clear, non-silent interruption to the user (per the no-fallback rule) rather than quietly truncating.
- _confidence: high_

#### [S1-7] `MEDIUM` · performance · effort S — Per-sample Data.append PCM encoding is a slow, allocation-heavy hot path for every recording/dictation buffer
- **Where:** shared/WaiComputerKit/Sources/WaiComputerKit/Audio/OpusEncoder.swift:31-54 (AudioEncoder.encode); called per buffer at MacRecordingViewModel.swift:456/478, iOS RecordingViewModel.swift:257/276, DictationSession.swift:188, RealtimePCMEncoder.swift
- **Problem:** encode() builds the int16 PCM by looping every sample and doing `withUnsafeBytes(of: intSample.littleEndian) { pcmData.append(contentsOf: bytes) }`. That is a closure invocation + a 2-byte `Data.append` per sample — for a 5-minute 16 kHz mono recording that's ~4.8M append calls, and it runs for every captured buffer on the recording/dictation path (and again inside RealtimePCMEncoder after resampling). It needlessly increases CPU/battery and adds GC-like churn on real-time-adjacent threads.
- **Fix:** Write into a preallocated buffer: allocate `Data(count: frameLength * channels * 2)` once and fill via a typed `UnsafeMutableBufferPointer<Int16>` (or build an `[Int16]` and wrap it), instead of per-sample append. Clamp + Int16 conversion stay the same; this is a drop-in micro-rewrite with big constant-factor wins.
- _confidence: high_

#### [S1-8] `MEDIUM` · logic · effort M — PCMResampler.convert() calls converter.reset() on every buffer, discarding FIR state and risking boundary artifacts
- **Where:** shared/WaiComputerKit/Sources/WaiComputerKit/Audio/PCMResampler.swift:54-66 (reset() inside convert)
- **Problem:** The class's whole rationale (docstring :7-14) is high-quality anti-aliased resampling for Russian fricatives. But convert() does `converter.reset()` before each single-buffer conversion and feeds endOfStream after one buffer. Resetting clears the sample-rate converter's internal filter history between consecutive buffers of the SAME continuous stream, so the FIR can't maintain continuity across buffer boundaries — producing small discontinuities/clicks every ~128ms (2048-frame buffers) precisely at the band edges the design is trying to protect. For one-shot conversions this is fine; for the continuous capture stream it partially defeats the purpose.
- **Fix:** Don't reset per buffer for a continuous stream — keep one converter alive per source format and feed buffers incrementally (only reset when the source format actually changes or the stream ends). If single-buffer isolation is required for correctness elsewhere, document why and measure the artifact tradeoff. Existing PCMResamplerReuseTests can be extended to assert continuity.
- _confidence: medium_

#### [S1-9] `MEDIUM` · logic · effort S — AudioEngineHost.copy(buffer:) returns the live aliased buffer on allocation failure — silent PCM corruption (no-fallback violation)
- **Where:** shared/WaiComputerKit/Sources/WaiComputerKit/Audio/AudioEngineHost.swift:227-250 (copy), specifically the `return buffer` at :230
- **Problem:** The tap callback copies each buffer because 'AVAudioEngine reuses the storage so we must convert immediately' (:84-86). But if AVAudioPCMBuffer allocation fails, copy() returns the ORIGINAL buffer, which is then appended to the pre-roll ring and yielded to the live stream. Because the engine immediately overwrites that storage for the next callback, the consumer reads scrambled/garbage samples — a silent data-corruption fallback. This contradicts the project's explicit 'no silent degradation — surface the error' rule.
- **Fix:** On allocation failure, drop the buffer (return nil and skip) and record a one-shot Sentry breadcrumb/metric instead of returning the aliased live buffer. Update handleTapBuffer to tolerate nil. Never hand back memory the engine is about to reuse.
- _confidence: high_

#### [S1-10] `MEDIUM` · security_privacy · effort S — Recording audio + transcript backups written to disk with no file protection / restrictive mode (raw audio is the most sensitive data)
- **Where:** shared/WaiComputerKit/Sources/WaiComputerKit/Recording/RecordingBackupStore.swift:230,602-605 (Data.write(.atomic), no protection); AudioFileWriter.swift:38-41 (header.write, FileHandle) — contrast SessionStore.swift:143 which sets 0o600
- **Problem:** Pending recordings persist the raw WAV audio (recording.wav), the diarised segments.json, and the transcript (manifest.json transcript field) under Application Support/WaiComputer/PendingTranscripts. None of these writes set NSFileProtection (iOS) or 0600 posix permissions (macOS). SessionStore deliberately sets 0o600 for the much smaller token file, but the far larger and more sensitive raw-audio/transcript corpus is written with default permissions and, on iOS, default (class C / NSFileProtectionCompleteUntilFirstUserAuthentication) protection rather than an explicit choice. For a 'second brain' this is the crown-jewel data at rest.
- **Fix:** On iOS set an explicit NSFileProtection level (at least completeUntilFirstUserAuthentication, ideally complete for audio) via write options / setResourceValues. On macOS apply 0o600 to the backup directory and files as SessionStore does. Make the protection level a single shared helper so all three artifacts are covered consistently.
- _confidence: medium_

#### [S1-11] `LOW` · code_quality · effort S — JSON websocket payloads are force-unwrapped (try!/!) — a non-encodable value crashes the streaming actor
- **Where:** shared/WaiComputerKit/Sources/WaiComputerKit/Network/WebSocketManager.swift:396-399 (encodeJSONPayload) and ProviderBackedRealtimeSession.swift:472-475 (encodeJSON)
- **Problem:** encodeJSONPayload/encodeJSON do `try! JSONSerialization.data(...)` then `String(data:encoding:.utf8)!`. The comment argues these are always-encodable static primitives, which is true today, but any future caller passing a non-JSON value (NaN Double, Date, a wrapped enum) turns a wire-protocol bug into a hard crash of the realtime actor mid-session. The project's no-fallback philosophy favors surfacing errors, not crashing the audio pipeline.
- **Fix:** These payloads are all tiny fixed dictionaries — build them as string literals (e.g. the control frames {"type":"Finalize"}/KeepAlive/CloseStream are constant) or have the function `throw` and let the existing send() error path handle it, instead of trapping. Keeps the streaming session resilient.
- _confidence: medium_

#### [S1-12] `LOW` · code_quality · effort S — WebSocketManager.connect() and reconnectLoop() build a brand-new default URLSession per attempt with no handshake delegate and no invalidation
- **Where:** shared/WaiComputerKit/Sources/WaiComputerKit/Network/WebSocketManager.swift:192-195 (connect) and :657-666 (reconnectLoop)
- **Problem:** Each connect/reconnect does `URLSession(configuration: .default)` and never invalidates it. Using URLSession.shared-style default config means the realtime WS shares the connection pool with Sparkle pollers/downloaders — the exact stall risk ProviderBackedRealtimeSession deliberately avoided by using an ephemeral, dedicated, delegate-bound session (see ProviderBackedRealtimeSession.swift:40-52 and its invalidateOwnedURLSession()). The created sessions are also not invalidated, so each reconnect leaks a session object until process exit.
- **Fix:** Give WebSocketManager one dedicated ephemeral URLSession (waitsForConnectivity=false, isolated from shared) created once, reuse it across reconnects, and invalidate on disconnect — mirroring ProviderBackedRealtimeSession. #### [S7-4] `HIGH` · logic · effort S — Buffer overflow / ArgumentException crash in WASAPI resampling accumulator (mic + system capture)

### Design system & visual craft (cross-cutting: web tokens/CSS, shared WaiComputerKit, macOS/iOS DesignSystem, Android theme, Windows/Linux styles)

_WaiComputer's web design system is genuinely strong: `web/src/styles/tokens.css` is a thoughtful, well-documented single source of truth (8pt spacing grid, 13-step type scale with editorial serif display, motion tokens, 3-step elevation, 7 WCAG-AA-tuned accents in light+dark, glass/material tiers) that consciously mirrors the macOS Swift `DesignSystem.swift`. macOS is a close second with a clever luminance-based `onAccentColor` for WCAG-safe text on any accent, and cached theme reads for scroll perf. The problem is that this rigor stops at the web↔macOS boundary. There is NO shared design-token source: every native platform re-declares spacing/typography/palette by hand, so they have drifted badly. The brand accent — the single most important identity signal — is amber on web/macOS but a DIFFERENT amber (#D17D2E vs #FF9500) on iOS, BLUE on Android (#2563EB) and Windows (#0E4DEA), and entirely absent on Linux (raw hex + Fluent defaults). iOS re-introduces the exact white-on-accent WCAG bug macOS already solved and ships zero Dynamic Type support. Apple platforms have no reduced-motion handling and no semantic danger/warning/success colors (raw `Color.red`). The iconography is also split: web ships the correct waveform-in-triangle mark while iOS renders a stale triangle-with-desktop-computer glyph (dead code that falsely claims parity). The path to premium consistency is to extract one cross-platform token spec and regenerate each platform's theme from it, then fix the per-platform a11y gaps._

### Product strategy (cross-cutting)

_WaiComputer's backend has built genuinely rare second-brain machinery — cited Ask-your-brain (/api/brain/ask), unified RRF hybrid search, compiled-truth dossiers, a bi-temporal EntityFact model, a nightly sleep-time consolidator with a governance gate, and a FastMCP agent server with OAuth + one-tap connect provisioning. The strategy docs (BRAIN_REDESIGN, BRAIN_SUPERPRODUCT) correctly identify cited memory recall + the MCP agent loop as the differentiators no incumbent (ChatGPT/Claude/Otter/Granola) can structurally match. BUT my code reading found the strategic core is profoundly under-shipped on clients and key moats are dead end-to-end: (1) the entire "Brain" flagship UI (cited Ask, dossiers) does NOT exist in web (no BrainPanel.tsx; "wai"/"library"/"content" routes all collapse to "inbox") and is NOT mounted in the current Mac app (no Brain feature folder, no .brain sidebar case) — so today the product users actually touch is a recordings-inbox + dictation tool, not a second brain; (2) the bi-temporal EntityFact moat is written by nobody (apply_fact_reconcile has zero callers) and read by nobody (brain_ask.py has zero EntityFact references) — built, advertised internally as the #1 lever, completely inert; (3) there is no proactive retention loop (no daily digest / resurfacing / nudge task in the Celery beat schedule — only billing reminders and an internal consolidator), so nothing pulls a user back daily; (4) the MCP moat is invisible (zero mentions on the landing page; the built one-tap connect provisioning isn't surfaced in the client, which still shows manual .cursor/mcp.json edits); (5) the value metric (weekly transcribed-word cap + a never-enforced 30-day memory window) prices the commoditizing capture layer, not the defensible memory/recall layer, and bills nothing today. There is also a large SPECS-vs-reality drift (SPECS sells a chat-first "Personal AI OS" with app-generation/deploy; reality is inbox-first with a static-HTML-template stub). The highest-leverage bets are about activating power that already exists, not building new capability._

### Round 2 — Performance & scalability

_A second, gap-filling performance pass focused on areas round 1 under-covered: the embedding write path, the recordings-only search SQL, the Brain graph/feed read paths, missing indexes on the hottest list query, web polling cost, and Android transcript rendering. The headline new findings are scale cliffs that are invisible on a small account but degrade linearly (or worse) as a user's corpus grows: (1) recording segments are embedded one-at-a-time with a serial network round-trip each (3 pipelines), adding minutes to processing and head-of-line-blocking the single Celery worker; (2) the recordings-only /search query computes its tsvector over a dynamically concatenated document and so cannot use the segment FTS GIN index — every search is a full seq scan with per-row JSONB aggregation; (3) the Brain graph + overview load pull the user's entire EntityMention/recording/item/chat corpus into Python on every open (the limit only trims output); (4) the Brain feed and since-last-seen endpoints each run a full three-source UNION with no per-source LIMIT; (5) the primary inbox list (also polled every 5s) has no composite (user_id, created_at) index so it seq-scans+sorts all of a user's recordings. All recommendations surface errors rather than masking them, consistent with the project's no-fallbacks rule. None of these duplicate the 120 round-1 findings (which covered the single worker, double ai_usage writes, no rate-limit on Ask/search, per-keystroke query embedding, wake_up EXISTS, Swift per-sample Data.append / Task-per-buffer, web single-bundle / 25-result cap, Android full-file readBytes / nested LazyColumn)._

- **Problem:** 1,000 transcribed words per week is roughly seven minutes of speech — a user can exhaust it in a single meeting, before the recall/search value lands. Best-in-class meeting tools (Fathom, tl;dv) lead with generous or unlimited free capture.
- **Fix:** Re-evaluate the free metric (higher weekly cap, or per-month) so first-week activation isn't throttled; if STT cost is the constraint, gate on *recall/Pro* features rather than capture volume.
- **Cross-ref:** Product strategy value-metric finding.

#### `HIGH` · ui_ux · effort M — Settings is one giant flat scroll (~14 sections; Danger Zone sits beside Appearance)
- **Where:** Settings (`audit/visual-1506/vis-08-settings.png`)
- **Problem:** Everything lives in a single endless scroll — Inbox & sources → Server & Data → Move-to-server → Export/Import → Telegram → Voice/Appearance → Identity & Voice → Transcription & summaries → Dictation → Account → Danger zone → Developer/MCP → Connected clients → API tokens. There is no tab or section navigation, so finding a setting means scrolling and scanning, and the destructive "Delete account" sits in the same flow as theme color.
- **Fix:** Group Settings into tabs/sections (the macOS app already uses a tabbed Settings — port that IA): e.g. Account · Capture & Transcription · Dictation · Appearance · Integrations (Telegram/MCP/API) · Data & Server · Danger zone. Isolate destructive actions.

#### `MEDIUM` · product · effort M — The Inbox is an undifferentiated firehose
- **Where:** Dashboard / Inbox (`vis-05-dashboard.png`)
- **Problem:** 50 mixed items — recordings, materials (PDF/article/YouTube), and Wai agent sessions — in one flat reverse-chron list, with several "Untitled material — needs input" (failed YouTube transcripts) clustered at the very top. There's no date grouping, no separation of "needs attention" from "ready," and the most valuable thing (a synthesized answer/overview) is absent. It reads as a queue, not a brain.
- **Fix:** Group by date/type, route the "needs input" items into their own tray rather than the top of the main feed, and lead with an answer/summary surface (the `BRAIN_REDESIGN.md` "Cards That Think" direction). Type filters (All/Recordings/Materials/Wai) already exist — make them more prominent and add date headers.

#### `MEDIUM` · ui_ux · effort S — Recording detail opens on the raw Transcript, not the Summary
- **Where:** recording detail (`vis-06-recording-detail.png` transcript wall vs `vis-07-recording-summary.png` summary)
- **Problem:** Opening a recording lands on the Transcript tab — a long single-column wall (a 47-min meeting is an enormous scroll). The Summary tab (well-built: overview + key points + action items) is the digestible value but is one click away and not the default.
- **Fix:** Default the detail to the Summary tab; keep Transcript one tap away. (Optionally collapse long transcripts by speaker turn / add a jump-to-section rail.)

#### `MEDIUM` · security_privacy · effort M — "Move to server" collects an SSH root password in a web form
- **Where:** Settings → Server & Data → "Move to server" (`vis-08-settings.png`)
- **Problem:** The self-host migration UI asks the user to type a VPS IP, SSH user (`root`), and SSH **password** directly into a browser form. Even for power users self-hosting their own box, prompting for a reusable root password in a web field is a security smell and an intimidating UX.
- **Fix:** Prefer an SSH-key / one-time bootstrap-token flow over a root password; never persist the password; add an explicit "advanced / self-host" gate with clear security guidance. (Verify intended threat model.)

#### `LOW` · product · effort S — The agent/MCP moat is well-built but buried in Settings (confirms "invisible, not missing")

But it lives at the bottom of a long Settings page, with nothing on the landing page or in onboarding pointing to it. The differentiator is real but undiscoverable.
- **Fix:** Surface "Connect your AI agent" as a first-class step in onboarding and a section on the landing page; the Settings UX itself is good — just promote it.
- **Cross-ref:** Product strategy (MCP moat invisible).
