# Android Parity Specification — WaiComputer

**Date:** 2026-04-17
**Owner:** Mik Wiseman
**Audience:** An implementing agent (new session, no prior context).
**Goal:** Bring the Android app to full functional parity with the iOS app, with a polished onboarding, optional-auth guest mode, live recording + transcription, library, and settings. Reference UX polish: `wai-money`, but refined further.

Do NOT implement anything outside what is described here. If ambiguity arises, stop and ask. Produce PR-sized commits keyed to the phases in section 11.

---

## 0. Non-negotiables

1. **Design-system consistency.** Material 3 + the visual tone set by `wai-money` (generous whitespace, rounded 16dp corners, soft neutrals, accent `#2563EB`, destructive `#DC2626`, warning `#F59E0B`). All shared spacing/typography tokens live in one file (`ui/theme/WaiTheme.kt`).
2. **One-tap record.** The `Record` tab must start recording with a single tap of the primary button. No intermediate screens or sheets. Mirrors the iOS fix from commit `1b40a683`.
3. **Offline-transcription banner.** If the ElevenLabs WebSocket cannot connect or drops mid-session, a visible orange banner surfaces under the top bar: *"Live transcription unavailable — audio is recording locally, transcript will be generated when you stop."* Same language as iOS/macOS.
4. **Local-first.** Audio is always written to disk before being sent over the network. Loss of network never loses the recording.
5. **Privacy-safe logging.** Never log raw transcript text, email addresses, tokens, filenames, search queries, or recording titles. Use `SentryHelper.sanitize(...)` equivalent as on iOS (`shared/WaiComputerKit/Sources/WaiComputerKit/Monitoring/SentryHelper.swift`).
6. **No fallbacks that hide errors.** Surface real failures with actionable messages.
7. **Same backend.** Host: `https://wai.computer`. All endpoints exist (iOS/macOS use them). Do not change backend.
8. **Testing gate.** `./gradlew testDebugUnitTest` + `./gradlew assembleDebug` must stay green. 80%+ coverage on new code.
9. **Bundle id / version.** Keep `applicationId = "is.waiwai.computer"`. Bump `versionCode` to 7 and `versionName` to `"1.1.0"` on first ship of this parity work. Reset Play Console release notes to reflect the new flow.
10. **Russian & English support.** All user-facing strings live in `res/values/strings.xml` + `res/values-ru/strings.xml`. The spec uses English as source; Russian copy is provided inline where flow-critical.

### 2.4 Guest mode (optional auth)
- "Try without account" creates **no** server identity. Recordings are saved ONLY locally (same backup format iOS uses).
- Guest can record, play back audio locally, see a transcript **only after they connect** (transcription requires backend). The Record UI clearly states: *"Transcribe and summarise after signing in"* with a lock badge on the transcript area.
- Post-record, the recording lives in `Library` with a `Local only` chip. Tapping it opens `GuestRecordingDetail`, which offers "Sign in to transcribe". On successful sign-in, `PendingSyncWorker` uploads all guest recordings via `uploadAudio` and they appear with full transcript/summary.
- Guest state is a single flag: `AuthStore.mode = Guest | Authenticated`. Never persist a fake user.
- Q&A tab is locked in guest mode with a "Sign in to ask questions about your recordings" empty state.
- Settings shows a prominent "Sign in to sync & transcribe" CTA at the top.

### 2.5 Authenticated user flow
- After login/register: brief success toast, navigate to `MainTab.Record`.
- `/me` is refreshed on every cold start and every app resume.
- 401 from any endpoint → try refresh; on refresh failure → clear tokens → route to `OnboardingAuthChoice` with a banner *"Session expired, please sign in."*

## 5. Auth subsystem

### 5.1 Backend endpoints (existing, do not modify)
| Method | Path | Body | Response |
|---|---|---|---|
| POST | `/api/auth/register` | `{email, password}` | `AuthResponse {access_token, refresh_token, user}` |
| POST | `/api/auth/login` | `{email, password}` | `AuthResponse` |
| POST | `/api/auth/magic-link` | `{email, client?}` | `MessageResponse` |
| POST | `/api/auth/verify-magic` | `{token}` | `AuthResponse` |
| POST | `/api/auth/refresh` | `{refresh_token}` | `AuthResponse` |
| POST | `/api/auth/logout` | `{refresh_token?}` | `MessageResponse` |
| GET | `/api/auth/me` | — | `UserResponse {id, email, created_at}` |

### 5.2 `AuthStore` (single source of truth)
- Backed by `SecureTokenStore` (access + refresh) and `DataStore<Preferences>` (mode + user id).
- Exposes a `StateFlow<AuthState>`:
  ```kotlin
  sealed interface AuthState {
      data object Unknown : AuthState         // initial, pre-check
      data object Onboarding : AuthState      // never signed in
      data class Guest(val since: Instant) : AuthState
      data class Authenticated(val user: UserSummary) : AuthState
      data class SessionExpired(val reason: String) : AuthState
  }
  ```
- Responsibilities:
  - `bootstrap()` — called once at app launch; loads tokens, calls `/me`, emits next state
  - `login(email, password)` / `register(...)` / `requestMagicLink(...)` / `verifyMagicLink(...)` — emit `Authenticated`
  - `continueAsGuest()` — emits `Guest(Instant.now())`
  - `logout()` — POST `/api/auth/logout`, clear tokens, emit `Onboarding`
  - `refresh()` — if a 401 comes back, single-flight refresh; on failure emit `SessionExpired`
- `WaiApi` asks `AuthStore` for the current access token via a suspending getter before each request. On `401`, `WaiApi` calls `AuthStore.refresh()` once, retries, or bubbles up.

### 5.3 `LoginScreen` / `RegisterScreen`
- Layout: logo + app name + subtitle, two tabs (Sign in / Create account), email + password, helper errors inline.
- Register: password min 8 chars, confirm password field, toggle show/hide.
- Validation rules:
  - email regex `^[A-Za-z0-9._%+\-]+@[A-Za-z0-9.\-]+\.[A-Za-z]{2,}$`
  - password ≥ 8 chars (server enforces)
- Inline error below field (Material3 `supportingText`).
- Global error snackbar for network / rate-limit (`429` → "Too many attempts, try again in a minute").
- Magic link option: "Prefer a sign-in link? Send to email" → `MagicLinkScreen` → enter email → toast "We sent a link to {email}". If the user opens the email's link on-device, our `MainActivity` handles the deep link `waicomputer://magic?token=...` and calls `verifyMagicLink`.
- Deep link registration: add `intent-filter` in `AndroidManifest.xml` matching `data android:scheme="waicomputer" android:host="magic"`.

### 5.4 Session expiry UX
- When `AuthStore` emits `SessionExpired`, `WaiAndroidApp` routes to `OnboardingAuthChoice` and shows a top banner: *"Session expired, please sign in again."* Banner persists until user logs in.

## 6. Recording subsystem

### 6.1 Audio capture — `AudioRecorder.kt`
- Uses `AudioRecord` with:
  - source: `MediaRecorder.AudioSource.VOICE_RECOGNITION` (tuned for speech)
  - sample rate: 16 000 Hz
  - channel: mono (`CHANNEL_IN_MONO`)
  - encoding: `ENCODING_PCM_16BIT`
  - buffer: max(2 × `AudioRecord.getMinBufferSize`, 2560 samples = 160ms @16kHz)
- Permission: `RECORD_AUDIO`. Request via `ActivityResultContracts.RequestPermission()` on first Record tap. If denied, show an inline CTA "Enable microphone in Settings" with a button that opens app settings via `Settings.ACTION_APPLICATION_DETAILS_SETTINGS`.
- Emits a `Flow<ShortArray>` of PCM frames (~100ms chunks). No conversion needed — we capture at the target rate directly.
- API:
  ```kotlin
  interface AudioRecorder {
      val isRecording: Boolean
      fun start(): Flow<ShortArray>      // cold flow; started on collect
      suspend fun stop()
  }
  ```

### 6.2 Local file writer — `AudioFileWriter.kt`
- Writes raw PCM frames to a WAV file in `filesDir/recordings/{recordingId}/audio.wav`.
- RIFF header finalised in `finalize()`.
- Fully thread-safe (single writer coroutine per recording).
- Matches the file format iOS uses so `uploadAudio` accepts it.

### 6.3 WebSocket manager — `WebSocketManager.kt`
Direct port of `shared/WaiComputerKit/Sources/WaiComputerKit/Network/WebSocketManager.swift` (lines 1–818). Use OkHttp 4.12 `WebSocket` (already a transitive dep of Ktor Android; add explicit `com.squareup.okhttp3:okhttp:4.12.0`).

- Config:
  - Endpoint: `wss://api.elevenlabs.io/v1/speech-to-text/realtime?model_id=...&token=...&include_timestamps=true&audio_format=pcm_16000&include_language_detection=true` (or `language_code=...` if not multi)
  - Token from `/api/transcription/session`, re-requested on each reconnect.
  - Commit strategy from session config.
- Events (Kotlin `sealed interface`):
  ```kotlin
  sealed interface WsEvent {
      data object Connected : WsEvent
      data class Transcript(val segment: LiveTranscriptSegment) : WsEvent
      data class Disconnected(val cause: Throwable?) : WsEvent
      data class Reconnecting(val attempt: Int, val max: Int) : WsEvent
      data object Reconnected : WsEvent
      data class ReconnectionFailed(val cause: Throwable?) : WsEvent
  }
  ```
- Reconnection: exponential backoff starting 500 ms, capped 30 s, jitter 0–50 %. Max 10 attempts. Buffer the last 30 s of audio chunks and replay on reconnect. End-of-stream commit chunk (640 zero bytes, `commit: true`) is deferred across reconnects just like iOS.
- Message parsing: handle `session_started`, `partial_transcript`, `committed_transcript`, `committed_transcript_with_timestamps`, documented error types (`auth_error`, `quota_exceeded`, `commit_throttled`, `unaccepted_terms`, `rate_limited`, `queue_overflow`, `resource_exhausted`, `session_time_limit_exceeded`, `input_error`, `chunk_size_exceeded`, `insufficient_audio_activity`, `transcriber_error`).
- Shape `LiveTranscriptSegment` identical to iOS: `{text, speaker?, is_final, start_ms, end_ms, confidence}`.
- Unit tests MUST cover: URL building, message parsing, reconnect buffering, end-of-stream deferring, error-type detection. Mirror `WebSocketManagerTests.swift` + `WebSocketManagerExtendedTests.swift` from iOS (21 tests total).

### 6.4 `RecordingViewModel`
- Phases: `Idle | Preparing | Recording | Finalizing`.
- Public state (all as `StateFlow`):
  ```kotlin
  data class UiState(
      val phase: Phase = Phase.Idle,
      val durationSeconds: Long = 0,
      val transcript: String = "",
      val connectionState: ConnectionState = ConnectionState.Connected,
      val liveTranscriptionOffline: Boolean = false,
      val isServerComplete: Boolean = false,
      val recordingType: RecordingType = RecordingType.Note,
      val error: String? = null,
  )
  ```
- `startRecording(...)`:
  1. Check mic permission
  2. Set `Preparing`
  3. If guest mode → skip the server calls, go straight to local-only flow (mic + file writer, no WS)
  4. Authenticated → `POST /api/recordings` → get recording id
  5. Start `WebSocketManager.connect()`; on failure set `liveTranscriptionOffline = true` and continue local-only
  6. Start mic flow → file writer + (if WS active) send chunks
  7. Set `Recording`
- `stopRecording()`:
  1. Set `Finalizing`
  2. Stop audio flow
  3. `WebSocketManager.finishStreaming()` — wait up to 5s for final segments
  4. If any final segments → `POST /api/recordings/{id}/live-transcript` with segments
  5. Else if WAV file exists → `POST /api/recordings/{id}/upload-audio` (multipart)
  6. On server failure → save to `LocalRecordingStore`; schedule `PendingSyncWorker`; show recovery banner
  7. Set `isServerComplete = true` on success; UI resets state

### 6.5 `RecordingScreen`
- One tap: bottom-centre pill button starts recording immediately.
- Pressing again stops.
- Same banner order as iOS; tests must render snapshots for: idle / preparing / recording / offline-banner / reconnecting / finalising / error / guest-mode.

### 6.6 Tests
- `RecordingViewModelTest` — MockApi + fake `AudioRecorder` + fake `WebSocketManager`; cover all 4 phases + offline path + save failure path + guest path.
- `WebSocketManagerTest` — MockWebServer with scripted messages; cover all event transitions; mirror iOS tests.
- `AudioFileWriterTest` — generate a WAV, parse back, assert sample count & header.
- `AudioRecorderTest` — run on device (instrumentation test) with a 1-second capture; assert flow emits non-empty frames.

## 8. Sync & local storage

### 8.1 `LocalRecordingStore`
Port of iOS `RecordingBackupStore`. Writes under `filesDir/recordings/{recordingId}/` with:
```
manifest.json          { recordingId, title, type, durationSeconds, transcript?, segments[]?, recordingType, createdAt, hasAudioFile }
audio.wav              (optional; present while audio is being captured or unresolved)
transcript.json        (optional; segments written after reconnection failures)
```

- `save(recording, ...)` — atomic write via `File.createTempFile` + rename.
- `recordSaveFailure(recordingId, message)` — marks manifest with `failureMessage`.
- `remove(recordingId)` — deletes directory.
- `listPending()` — returns all manifests marked as pending.
- Thread-safe via a `Mutex` per recording.

### 8.2 `PendingSyncWorker`
- `CoroutineWorker` scheduled via `WorkManager.enqueueUniqueWork(name="pendingRecordingSync", ExistingWorkPolicy.KEEP)`.
- Retry constraints: `NetworkType.CONNECTED`; exponential backoff 30s → 30min.
- Per pending recording:
  1. If audio file exists → `uploadAudio`
  2. Else if transcript.json exists → `saveLiveTranscript`
  3. On success → `LocalRecordingStore.remove(...)`
  4. On permanent failure → log + keep manifest (user can retry from detail screen)
- Triggered on: app resume, login success, network-available signal from `NetworkMonitor`.

### 8.3 Guest → authenticated migration
- On successful login/register while in guest mode:
  1. Toast "Migrating your recordings…"
  2. Enqueue `PendingSyncWorker`
  3. All guest recordings upload + transcribe in background; library updates as `ready` arrives
- If a guest declines to log in and later uninstalls, local data is lost. Document this in `GuestModeInfoSheet`: *"Guest recordings live only on this device. Sign in to keep them safe."*

## 13. Permissions & manifest changes

Add to `AndroidManifest.xml`:

```xml
<uses-permission android:name="android.permission.RECORD_AUDIO" />
<uses-permission android:name="android.permission.POST_NOTIFICATIONS" /> <!-- Android 13+ for sync worker foreground -->
<uses-permission android:name="android.permission.FOREGROUND_SERVICE" />
<uses-permission android:name="android.permission.FOREGROUND_SERVICE_MICROPHONE" />
```

Foreground service for active recording (so the OS doesn't kill the mic when user backgrounds the app): declare a `Service` that posts an ongoing notification "Recording in progress — tap to return to WaiComputer".

Deep-link intent-filter:
```xml
<intent-filter android:autoVerify="false">
    <action android:name="android.intent.action.VIEW" />
    <category android:name="android.intent.category.DEFAULT" />
    <category android:name="android.intent.category.BROWSABLE" />
    <data android:scheme="waicomputer" android:host="magic" />
</intent-filter>
```

Update `android:label="WaiComputer"` (current value `"WaiComputer"` is wrong) and pick production app icon from `ios/WaiComputer/WaiComputer/Assets.xcassets/AppIcon.appiconset/` — render at `mipmap-*dpi` densities via Android Studio's Image Asset wizard.
